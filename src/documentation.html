# 1. App Overview

## What the app does
Financial Foundation Check is a guided financial profile web app that helps users assess their current financial readiness, generate a scored "Financial Fortress" result, and request a personalized estimate follow-up.

## Who it is for
- Individuals who want a quick financial readiness check
- Users exploring protection, emergency fund, and goal planning options
- Leads who may want advisor guidance without immediate commitment

## Main business purpose
- Educate and qualify users through a short structured journey
- Convert completed assessments into advisor-ready estimate requests
- Capture contact intent data with consent

## Main user outcome
- User gets a personalized readiness score, category, and recommendations
- User can submit details for a free personalized review

---

# 2. UI/UX Flow

## End-to-end journey
1. Welcome screen introduces the experience and prompts user to start.
2. Hub screen shows module roadmap and progression.
3. Module screens collect profile answers in sequence (single and multi-select questions).
4. Analyze transition screen displays a short "building" animation/message.
5. Dashboard screen presents score, persona, layer analysis, and CTA actions.
6. Quote transition screen explains the next step before data collection.
7. Multi-step quote form captures profile and contact details.
8. Quote teaser confirmation screen summarizes request status and next steps.

## UX patterns used
- Guided flow with progressive disclosure
- Locked/unlocked modules based on completion order
- Animated feedback on completion and transitions
- Sticky CTA behavior in dashboard with observer-based collision handling
- Validation-based continue buttons for all forms

# 5. Business Logic

## Progression rules
- Modules follow ordered unlocking: next module unlocks only after previous completion
- Final results CTA appears only when all modules are completed and readiness flag is set
- Quote form progresses in fixed sequence with per-step validation

## Scoring model
- Total score aggregates category breakdowns:
  - Cash flow
  - Emergency
  - Protection
  - Goals
- Category values are computed from selected answers
- Persona and ring color are derived from score thresholds

## Persona outcomes
- High score: strong foundation persona
- Mid score: on-track persona
- Lower score: building phase persona

## Threat generation logic
- Emergency fund level drives unexpected expense risk messaging
- Health coverage presence/absence drives health risk messaging
- Dependents + missing life cover can trigger family backup warning

## CTA generation logic
CTA text and focus change based on user profile combinations, including:
- Dependents and missing life protection
- No protection selected
- Investment priority with low resilience score
- Default momentum-building path

## Observer-based layout logic
- Dashboard sticky CTA visibility is controlled by an IntersectionObserver
- When inline dark CTA enters viewport, sticky CTA slides out and disables pointer events
- When inline CTA leaves viewport, sticky CTA slides back in

## Submission behavior
- Quote final step triggers confirmation state and success animation
- No backend persistence is visible in current code; behavior is in-memory UI flow

# 9. Conditions and Edge Cases

## Disabled and gated states
- Continue buttons disabled until required input is present
- Locked module cards are non-interactive
- Final submit is blocked until consent is checked

## Completed states
- Module completion triggers visual confirmation effects
- Completed modules show check state and timing message
- Final module completion unlocks analyze action after transition timing

## Conditional messaging
- Mini insight appears at specific partial completion points
- Threat cards change by profile conditions
- CTA headline/hook/button text changes by detected user needs

## Empty states
- Before any module completion, hub shows roadmap with locked future modules
- Optional sections (breakdown accordion) can remain collapsed with clean default view

## Error handling visibility
- No explicit server/API error states observed (no network submission in current code)
- Primary safeguards are form validation and disabled actions

## Transitions and redirects
- All transitions are state-driven inside a single-page flow
- Timed transitions used for analyzing and celebration phases
- Restart function clears all in-memory state and returns to welcome

# 10. Manual Agent Configuration

## Scope
The multi-agent assessment app should remain stateless. Its responsibility is to load the agent configuration, accept the form submission, email the configured agent, and forward the same payload to the external system. Lead storage, recommendation processing, pipeline status, and history belong to the external system.

## URL pattern
- https://assess.lablibre.com
- https://assess.lablibre.com/advisor
- https://assess.lablibre.com/team-member
- https://assess.lablibre.com/agent_name

The first path segment is the agent slug. Unknown or inactive slugs should show an unavailable page instead of accepting submissions. The root URL can resolve to an active configured agent when DEFAULT_AGENT_SLUG is set to that agent's slug.

## Required agent fields
```json
{
  "slug": "advisor",
  "agentName": "Advisor Name",
  "agentEmail": "advisor@example.com",
  "toolName": "Financial Foundation Check",
  "headline": "Check your financial foundation in minutes",
  "subheadline": "Answer a few questions and get a personalized readiness snapshot.",
  "status": "active",
  "externalSystemEndpoint": "https://external-system.example.com/api/assessment-intake"
}
```

## Manual setup checklist
1. Add the agent configuration record in the configured agent source.
2. Keep the slug lowercase, unique, URL-safe, and clear of reserved paths such as api or asset routes.
3. Set agentEmail to the advisor inbox that should receive the complete assessment submission.
4. Confirm EMAIL_FROM is a shared verified sender on the Cloudflare domain.
5. If the root URL should use one advisor profile, set DEFAULT_AGENT_SLUG to that active slug.
6. Keep external API secrets server-side only.
7. Open /{slug} and confirm the agent-specific tool name and copy render correctly.
8. Submit a test assessment and verify the email, Reply-To, and external payload forwarding.

## Recommended email headers
```txt
From: Financial Foundation Check 
To: Agent Name 
Reply-To: Lead Name 
Subject: New Financial Foundation Lead: Lead Name
```
